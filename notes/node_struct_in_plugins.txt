This confusion is completely normal — because VPP uses three related but different structures.

You are mixing:

1️⃣ vlib_node_registration_t
2️⃣ vlib_node_t
3️⃣ vlib_node_runtime_t

They are NOT the same.
They exist at different stages.

Let’s clear this permanently.

🧠 Big Picture First

Think of building a factory machine.

Stage	Real Life Example
Registration	Blueprint
Node	Installed Machine
Runtime	Machine running on each worker

That’s exactly what VPP does.

1️⃣ vlib_node_registration_t
(Blueprint / Template)

Created by:

VLIB_REGISTER_NODE(packet_count_node)

This structure:

Exists in ELF file

Is static metadata

Describes the node

Is NOT used for packet processing

It contains:

name
vector_size
type
next node names (strings)
function pointer

It is stored in:

ELF section (.vlib_node_registration)

It exists before VPP even starts.

It is only used during startup.

2️⃣ vlib_node_t
(Installed Machine in Graph)

When VPP starts:

It reads all vlib_node_registration_t structures
Then creates real graph nodes:

vlib_node_t

This is the real node stored in:

vlib_main.node_main.nodes pool

It contains:

Node index

Name

Function pointer

Next node indices (NOT strings anymore)

Type

Vector size

This is permanent for entire runtime.

This is what show node displays.

3️⃣ vlib_node_runtime_t
(Machine Instance Per Worker Thread)

VPP is multi-threaded.

Each worker thread gets its own optimized copy of runtime info.

So for every worker:

vlib_node_runtime_t

is created.

Stored in:

vlib_main.node_main.node_runtime[thread_id]

This contains:

Fast next node indices

Cached counters

State flags

Runtime scheduling data

This is what show runtime reflects.

🔥 Now Let’s Connect Them

Startup flow:

Plugin loads
    ↓
Find vlib_node_registration_t (blueprint)
    ↓
Create vlib_node_t (real graph node)
    ↓
For each worker:
    create vlib_node_runtime_t

So order is:

registration → node → runtime
🎯 Clear Differences
Structure	Created When	Stored Where	Purpose
vlib_node_registration_t	Compile time	ELF section	Template
vlib_node_t	Startup	Node pool	Graph definition
vlib_node_runtime_t	Worker start	Per-thread runtime pool	Execution state
🧠 Why 3 Structures?

Because VPP is designed for:

Dynamic plugin loading

Graph construction after all nodes known

Multi-threaded execution

Maximum performance

If only one structure existed:

It would mix configuration and runtime

Hard to scale per thread

Hard to resolve dependencies

So VPP separates concerns cleanly.

🔬 Visual Memory Layout

After VPP starts:

VPP Process Memory

ELF section
  └── vlib_node_registration_t (used only at startup)

Node pool
  └── vlib_node_t (one per node)

Worker 0
  └── vlib_node_runtime_t for packet_count

Worker 1
  └── vlib_node_runtime_t for packet_count

Worker 2
  └── vlib_node_runtime_t for packet_count
🧵 Why Runtime Structure Is Needed?

Because:

Workers run in parallel

Each worker needs separate stats

Each worker has its own frame queue

Each worker must avoid locks

So runtime structure is per-thread.

🧠 In Simple Terms

If your node is:

packet_count

Then:

Registration = Description of packet_count

Node = packet_count exists in graph

Runtime = packet_count running on worker 0

Runtime = packet_count running on worker 1
