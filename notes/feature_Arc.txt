🧠 First: What Is a Feature Arc?

VPP packet processing is a graph:

tap-input
   ↓
ethernet-input
   ↓
ip4-input
   ↓
ip4-lookup
   ↓
ip4-rewrite
   ↓
interface-output

Now VPP allows plugins to “hook into” specific stages.

Those hook points are called:

🔥 Feature Arcs

A feature arc is:

A dynamic chain of optional nodes inserted at a specific graph stage.

🧠 What Is ip4-unicast Arc?

ip4-unicast is the feature arc used during IPv4 forwarding.

It runs between:

ip4-input
    ↓
[ip4-unicast feature arc]
    ↓
ip4-lookup

So when you attach to ip4-unicast, your node runs:

Only for IPv4 packets

During unicast forwarding

Before routing decision

🧠 What Does VNET_FEATURE_INIT Do?

This macro:

VNET_FEATURE_INIT(packet_count_feature, static)

creates a structure:

vnet_feature_registration_t

This structure is placed in a special ELF section:

.vnet_feature_registration

Just like nodes use .vlib_node_registration.

🧠 What Happens at Startup?

During VPP initialization:

Plugin is loaded

All feature registration structs are scanned

VPP builds feature arc lists

Feature ordering is resolved

But feature is NOT enabled yet

Important:

This macro does NOT enable your feature.

It only registers it as available.

🔹 .arc_name = "ip4-unicast"

Internally:

VPP has a table:

feature_main.arc_index_by_name["ip4-unicast"]

This string is converted to:

arc_index (integer)

From that moment on:

Your feature is associated with that arc.

VPP converts:

So it now knows which graph node to execute.

🔹 .runs_before = VNET_FEATURES("ip4-lookup")

This is the most powerful part.

Feature arcs are ordered chains.

Multiple plugins may attach to same arc.

Example:

ACL
NAT
IPSEC
Your Plugin

Who runs first?

Ordering is resolved using:

.runs_before

.runs_after

You specified:

So VPP builds dependency graph and sorts features accordingly.

Internally it performs a topological sort.

🧠 How Ordering Is Resolved

Suppose arc has:

featureA runs_before featureB
featureB runs_before featureC

VPP builds:

featureA → featureB → featureC

It ensures correct execution order.

If conflict occurs → VPP throws error at startup.

🧠 Important: Feature Registration vs Enabling

Your macro only registers.

🧠 What Happens When Feature Is Enabled?

When you run:

set interface feature tap0 ip4-unicast packet_count

VPP:

Finds interface index

Finds arc index

Builds per-interface feature config

Rewrites graph next pointers

Inserts your node into path

Now packet path becomes:

ip4-input
    ↓
packet_count
    ↓
ip4-lookup

Only for that interface.

🧠 Where Is Feature Configuration Stored?

Feature system uses:

vnet_feature_config_main_t

Per interface:

interface_feature_config[interface_index]

Each interface has its own feature chain.

That’s why:

show interface features tap0

works per-interface.

🧠 Runtime Execution Flow

When IPv4 packet arrives:

ip4-input
    ↓
vnet_feature_arc_start(arc_index)
    ↓
feature_node_1
    ↓
feature_node_2
    ↓
ip4-lookup

Dispatcher executes feature nodes in resolved order.

Your node is just one element in this chain.

🧠 Internal Data Structures

At runtime:

feature_main
   ├── arcs[]
   │      ├── arc name
   │      ├── feature registrations
   │      ├── ordered feature list

Each interface holds:

feature_config[interface]
   ├── arc_index
   ├── feature_bitmap
   ├── next_node_indices

So enabling feature modifies bitmap.

Per-interface control

Ordered dependency resolution

Lock-free execution

Fast index-based dispatch

No string lookups.

Everything precomputed.
