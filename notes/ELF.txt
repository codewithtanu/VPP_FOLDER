🧠 What Is an ELF File?

ELF = Executable and Linkable Format

It is the standard binary format used by Linux for:

Executables (vpp)

Shared libraries (.so)

Object files (.o)

Kernel modules (.ko)

So when you compile your VPP plugin:

make build

The output:

packet_counter_plugin.so

is an ELF shared object file.

🏗️ When Is ELF Created?

ELF files are created during compilation and linking.

Compilation stages:
1️⃣ .c file
2️⃣ compiled to .o (ELF relocatable object)
3️⃣ linked into .so (ELF shared object)

Example:

gcc -c packet_counter.c

Creates:

packet_counter.o   ← ELF object file

Then:

gcc -shared packet_counter.o -o packet_counter_plugin.so

Creates:

packet_counter_plugin.so   ← ELF shared library

So:

👉 .o files are ELF
👉 .so files are ELF
👉 vpp binary itself is ELF

📂 Where Is It Stored?

For VPP plugins, typically:

build-root/install-vpp-native/vpp/lib/x86_64-linux-gnu/vpp_plugins/

Or system install:

/usr/lib/x86_64-linux-gnu/vpp_plugins/

Your plugin .so lives there.

🔬 What Is Inside an ELF File?

An ELF file is NOT just machine code.

It contains structured sections.

Main parts:

ELF Header
Program Headers
Section Headers
Sections (.text, .data, .bss, etc.)
Symbol Table
Relocation Info

Let’s break that down.

📦 Important ELF Sections
1️⃣ .text

Stores your compiled machine code.

Example:

packet_counter_node_fn()

show_packet_counter()

This is executable memory.

2️⃣ .data

Stores initialized global variables.

Example:

packet_counter_t counter = {0};

This lives in .data.

3️⃣ .bss

Stores uninitialized globals.

Example:

packet_counter_t counter;

Allocated but initialized to zero at runtime.

4️⃣ .rodata

Read-only data:

.description = "Simple protocol counter"

Stored here.

5️⃣ .symtab

Symbol table.

Stores names of:

Functions

Global variables

Node registration symbols

VLIB_PLUGIN_REGISTER symbol

VPP uses this to locate your plugin registration structure.

🧠 How Linux Loads ELF

When VPP starts and calls:

dlopen("packet_counter_plugin.so")

Linux dynamic loader:

Reads ELF header

Maps .text into executable memory

Maps .data into writable memory

Zeroes .bss

Resolves external symbols

Fixes relocation entries

Calls constructors (if any)

Now plugin is live in memory.

📜 What Is ELF Header?

At the beginning of every ELF file:

0x7F  'E'  'L'  'F'

You can verify:

readelf -h packet_counter_plugin.so

You’ll see:

Architecture (x86_64)

File type (Shared object)

Entry point

Section offsets

🔥 Types of ELF Files
Type	Example
Relocatable	.o
Executable	vpp
Shared Object	.so
Core Dump	core file

Your plugin = Shared Object ELF.

🧵 How Does This Relate to Your Macros?

When you write:

VLIB_PLUGIN_REGISTER() = {...};

That structure is placed into a special ELF section.

During dlopen, VPP finds that section via symbol lookup.

So ELF makes dynamic plugin system possible.

Without ELF → no runtime plugin loading.

🧠 Memory Layout After Loading

When loaded:

VPP Process Memory
 ├── VPP main code (.text)
 ├── Plugin .text
 ├── Plugin .data
 ├── Plugin .bss
 ├── Heap
 ├── Buffer pools

All inside one process.

🔍 You Can Inspect Your Plugin

Run:

readelf -S packet_counter_plugin.so

Shows sections.

Run:

nm packet_counter_plugin.so

Shows symbols.

Run:

objdump -d packet_counter_plugin.so

Shows disassembled machine code.

🧠 Important Concept

ELF is just a container format.

It organizes:

Code

Data

Metadata

Relocation info

So OS can load and execute it correctly.

🎯 In Simple Terms

Think of ELF as:

A structured box containing your compiled code, variables, and metadata so Linux knows how to load it into memory.

🚀 Now Let’s Connect This Back to VPP

Your plugin exists because:

Compiler created ELF .so

VPP used dlopen

Linux loader mapped ELF sections

VPP registered nodes using symbols from ELF

No ELF → No plugin system.
