🧠 First: What Is a Worker Thread in VPP?

VPP is a multi-threaded packet processing engine.

It creates:

1️⃣ Main thread  
2️⃣ One or more Worker threads  

Each worker thread runs on a CPU core (lcore).

You can check:

show threads

Example output:

Thread 0 (main)
Thread 1 (worker-0)
Thread 2 (worker-1)
Thread 3 (worker-2)

Each worker is pinned to a CPU core.

🔥 What Does a Worker Thread Do?

Each worker runs the VPP graph dispatcher loop.

Simplified:

while (1)
{
   get next frame
   call node function
   forward to next node
}

Each worker independently processes packets.

🎯 Now Important Question

Do all nodes process in different lcores?

The answer:

👉 Each worker has its own copy of all nodes.
👉 A packet is processed entirely by ONE worker thread.
👉 Nodes are NOT split across cores for a single packet.

🧠 How It Works Conceptually

Imagine:

You have 4 CPU cores.

VPP creates:

Worker 0 → Core 1
Worker 1 → Core 2
Worker 2 → Core 3
Worker 3 → Core 4

Now incoming packets are distributed across workers.

Example:

Packet A → Worker 0
Packet B → Worker 1
Packet C → Worker 2
Packet D → Worker 3

Each worker processes its own packets fully through the graph.

🔥 Very Important

A single packet path:

tap-input
   ↓
ethernet-input
   ↓
ip4-input
   ↓
packet_count
   ↓
ip4-lookup

All of that happens on the SAME worker thread.

It never jumps between cores.

🧠 Why?

Because switching cores would require:

Locks

Synchronization

Cache invalidation

Huge performance penalty

So VPP design is:

Packet stickiness to one worker
🧵 Where Do Workers Come From?

If using DPDK:

Each RX queue is assigned to a worker

NIC RSS distributes flows to queues

Each queue is polled by one worker

If using TAP/AF_PACKET:

Software distributes packets to workers

🔥 What Is lcore?

In DPDK terminology:

lcore = logical CPU core

VPP worker threads are pinned to lcores.

Example:

taskset -p <vpp_pid>

You can see CPU affinity.

🧠 How Many Copies of Node Exist?

Important:

vlib_node_t → one global definition

vlib_node_runtime_t → one per worker

So if you have 4 workers:

packet_count runtime for worker 0
packet_count runtime for worker 1
packet_count runtime for worker 2
packet_count runtime for worker 3

Each worker executes its own runtime structure.

🔥 Why Your Global Counters Are Wrong

Because:

total_packets++;

If 4 workers increment simultaneously:

Race condition.

Correct design:

Per-thread counters:

u32 thread_index = vm->thread_index;

Then:

counter[thread_index]++;

Later aggregate in CLI.

🎯 So Final Answer

Do all nodes process in different lcores?

No.

Correct understanding:

Each worker runs entire graph independently.

Packets are distributed across workers.

Each packet stays on one worker.

Nodes are replicated logically per worker (runtime structure).

This model gives:

No locks

No contention

Per-core cache locality

Extremely high throughput

That’s why VPP scales linearly with cores.
